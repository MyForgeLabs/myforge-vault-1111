    6. Auth + RBAC#

    • Két párhuzamos auth-rendszer:
    • Admin-auth (packages/core/auth) — klasszikus email+jelszó JWT, refresh-token cookie (httpOnly), Role enum 8 szint: OPERATOR(1) / TECHNIKUS(2) / BOLTVEZETO(3) / ACCOUNTANT(3) / PARTNER_OWNER(4) / CENTRAL_ADMIN(5) / DEVOPS_ADMIN(6) / SUPER_ADMIN(8). PIN-login is van pultos-flowhoz. Endpoint-ok: /auth/login, /refresh, /logout, /logout-all, /pin-login, /forgot-password, /reset-password, /verify-password.
    • Portal-auth (apps/kgc-api/src/modules/portal-auth) — jelszó NÉLKÜLI, SMS OTP + Magic-link email. Külön PortalUser tábla, külön PortalAuthGuard, req.portalUser.{id,tenantId,partnerId} szemantika.
    • Multi-tenancy: PostgreSQL Row Level Security (RLS) policies (prisma/policies/rls-policies.sql). RlsService + TenantContextInterceptor automatikusan injektálja a tenant_id-t Prisma middleware-en keresztül — a findMany({where:{tenantId}}) manuális szűrés TILTOTT (AGENTS.md "Critical Rule #1"). Schema-szinten minden tábla tenantId: Uuid és RLS-policy van rá.
    • 6 RLS-batch migráció Sprint 24 W3 körül landed: batch1_rls_auth → batch6_rls_integration + 2026-05-15 e107_rls_policies_consolidation. A BYPASS_RLS mód BypassRlsPrismaService-en keresztül opt-in (admin tooling).
    • API rate-limit: ApiRateLimitGuard (Redis-alapú, in-memory fallback) globálisan; publikus endpoint-ok külön throttle (5/min, 3/h).
    • Külső API auth (kgc-berles → kgc-api): nincs külön API-key infrastruktúra. Két reális opció:
    • portal-equipment style anonymous + tenantId-query (ami most már megvan és működik) — a Next.js BFF/API-route ezt simán kihasználhatja.
    • Új service-account JWT vagy TenantSecret (packages/core/tenant/src/prisma-extensions + tenant-secret.prisma table) — ADR-112 már encrypted per-tenant API-keys-t használ provider-integrációkhoz, ezt lehet bővíteni.

    7. BMAD Phase 5 = mi#

    A BMAD = "Business-driven Multi-AI Development" — Zoli saját agentic workflow-rendszere, ami a _bmad/ mappában tárol konfigot (bmm, cis, tea, wds, bmb sub-module-ok, config.toml, scripts), és a _bmad-output/-ba teszi az artifact-eket (sprint-retro-k, ATDD checklist-ek, brainstorming, AI cross-review-k).

    Fázisok eddig: - Phase 1-4 = a tényleges feature-build (Sprint 1-25, ~80 epic, ~470 story implemented). - Phase 5 = "Tesztelési Fázis" = stabilizáció + foundation-hardening: - Test-coverage emelés (vitest unit + Playwright E2E + security-fuzz) - RLS-policy consolidation (E-107) - CREATE TYPE idempotency wrap (Phase 6 + 6.1 + 6.2 — 136 CREATE TYPE statement idempotensé tétel) - Q-T quality tracker tasks (Q-T008..Q-T014) = adversarial review-ban talált P0/P1 hibák (pl. Q-T012 = PayslipDispatch silent-failure-loop a production-ben — BullMQ.email queue szennyezés miatt soha nem ment el bérlapfizetési email, fix queue-isolation-nel) - "Murat" middle-way pathway = adversarial review-ban talált hibákra batch-stratégia (Murat strategy A/B/C/D — smoke-healing, baseline-fix, cross-module scaffold, type-safety) - "Foundation closure" = a foundation-layer (DB migration + RLS + auth) blokkoló-mentesítése a következő nagy feature-sprint előtt - Sprint 26 W0 (most lezárva, 2026-05-18): 7.9 SP, 6 commit, Phase 6/6.1/6.2 complete + Mary-prep ready. - Sprint 26 W1 (D-3-ban most): Q-T008→Q-T012 tracker-cleanup, Murat A+B+C+D batchek, payslip-fix, e2e PCJ-11 cross-tenant status-code fix. Várja a "Mary mini-discovery"-t. - "Mary discovery" = a Mary egy BMAD persona (analyst/discovery-role, vélhetően a BMAD bmm-modul agent-je) — a "Szerviz PM #2 scope-discovery kliens-mini-egyeztetés" mini-discovery prep szerint 4 kérdéses kliens-egyeztetés Javo/Zsuzsi-val a Szerviz-modul gap-finding-jére. - Várható Sprint 26 W2+: Mary-call-driven Service slice batch (~3-10 SP).

    Ami HIÁNYZIK (a kgc-berles épülésekor érintheted Zolival):

    1. Cikk / CMS / blog modul: NINCS. Sem Article, Post, Content, Page model nem létezik. Ha a Next.js site-on cikkeket akarsz, az NEM jön a KGC-4-ből — vagy külön headless CMS (Sanity/Strapi/Payload), vagy a Next.js-ben fájlrendszer-alapú MDX kell.
    2. EquipmentType még nem szerepel a portal-equipment API-ban. A portal-equipment a "régi" RentalEquipment-en operál, NEM az új EquipmentType katalóguson (E-123 új tábla, heroImageUrl + galleryImages[] mezőkkel). Ezt kérni kéne, mert a webshop-face képek és gallery oda kerülnek.
    3. Termék-image / Equipment-image: a portal-equipment most NEM ad vissza képet. A schema-ban van EquipmentType.heroImageUrl és Product.images (feltehetően), de a PortalEquipmentListItem interface csak: id, name, category, brand, dailyRate, weeklyRate, availability. Bővítendő API.
    4. SSO a két frontend között (admin ↔ public site): nincs cross-app cookie kontraktus most. Ha "MyKGC" portal-userrel SSO-t akarsz a kgc-berles felé, vagy a portal-JWT-t kell shared-secret-szel verify-elni Next-en, vagy NextAuth-szal teljesen külön session.
    5. Real-time / WebSocket: apps/kgc-api/src/modules/dashboard/websocket/ van (admin-side dashboard events), de portal-side notification push NINCS — a notification és portal-notifications modulok REST-pollos.
    6. Public catalog feed (CSV/JSON sitemap): nincs export-endpoint a webshop-szinkronhoz. Ha SEO-szinten egy /products.json feed kell, kérni kell.
    7. Versioning: /api/v1 van, de a portal-route-okon NINCS verzió-prefix (Swagger doc szerint /portal/auth/... — valójában a global-prefix miatt /api/v1/portal/...). Tisztázandó.

    Javaslat a kgc-berles architektúrára: - Next.js (:3004) → BFF route-okon → KGC-4 API (/api/v1/portal/* és /api/v1/public/*). - A tenantId legyen env-konstans a Next-app .env-ben (KGC_TENANT_ID=...), NEM URL-paraméter. - Authentikációhoz: indulj anonymous flow-val (csak portal-equipment + public-inquiry + public-service-request), és opt-in OTP/Magic-link → /portal/bookings POST a foglalás-flowhoz. - Image-eket: amíg portal-equipment nem adja vissza a heroImageUrl-t, addig közvetlenül a Product/EquipmentType endpoint kell (vagy bővítsd a PortalEquipmentService.list/detail select-et — apró PR Zoli-nak). - A [[11-wiki/nextjs-api-proxy-bridge]] wiki-mintával csináld — egyezik a balance/example-balance.local stílusoddal.

    3. Termék-képek a portal-API-ban: a portal-equipment.service select jelenleg NEM adja vissza a Product.images / EquipmentType.heroImageUrl-t — Zoli-kérés: bővítse a select-et, vagy adja meg az URL-konvenciót (MinIO publikus bucket? CDN-front?).
    4. tenantId átadás a publikus API-n — most query-string. Production-ben kell-e Origin-alapú resolve (mint a public-inquiry-ben)? Mert ha kgc-berles :3004 is más originról jön mint a kgc-portal, akkor a CORS_ORIGINS env-be is fel kell venni + tenant-resolution stratégiát egyértelműsíteni.
    5. API-versioning a portal-routes-on: /api/v1/portal/* lesz a tényleges URL (global-prefix miatt) — confirmed.
    6. EquipmentType vs RentalEquipment — két párhuzamos katalógus-modell. A "webshop-face" (E-123 STORY-1306 D5) EquipmentType.heroImageUrl-be kerül, de a portal-equipment még a régi RentalEquipment-tel dolgozik. Mi a roadmap a két modell egyesítésére / a portal-API E-123-as átállására?
    7. Cikk / CMS modul: nincs. Megerősítendő, hogy ezt a kgc-berles oldalán külön kezeljük (MDX / Sanity / Payload). Ha integrálni akarjuk a KGC-4-be, az egy új epic-feature.
    8. MyPOS / payment-callback flow: a kód szerint még stub-szinten. Production-ben a public site fizetés-redirect csak akkor működik, ha ez bekapcsolódik.
    9. apps/kgc-portal — már létezik egy portal-frontend app. Mi a viszonya a tervezett kgc-berles-hez? Külön landing-page lesz csak (kgc-berles = marketing-site), vagy a portal-flow-t is meghívja?
    10. Rate-limit a portal-equipment endpointokon: NINCS @Throttle decorator! Csak a globális ApiRateLimitGuard védi. A public-inquiry 5/min/IP-t kapott, a public-service-request 3/h-t — a portal-equipment list/detail nyitva van. SEO-bot vs adversarial bot ellen kérni kell hardenelést.
